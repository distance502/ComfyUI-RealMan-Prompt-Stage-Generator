# 随机主题池风格模式图片测试与优化

## 背景

当前节点已经具备两套会同时影响出图差异的机制：

1. `模板风格` 会通过 `stage_prompt/prompt_builder.py` 中的 `style_lead`、`style_tags`、`extra_fragments` 注入风格导向。
2. `运行时随机标签` 会通过 `stage_prompt/randomizer.py` 与 `_RUNTIME_DIVERSITY_VARIANT_PROFILES` 改写主体、场景与局部语义。

这两套机制叠加后，用户在观察不同风格模式出图时，容易看到的是“题材变化”而不是“风格变化”。当前缺少一组固定主题、只切换风格变量的可复跑测试，因此无法稳定判断以下问题：

- 不同 `模板风格` 的提示词分化是否足够明确。
- 风格 profile 的差异是否真的转化为可见的图像风格差异。
- 随机主题池是否正在稀释风格模式本应体现的分化。

## 目标

本次工作只解决一个核心目标：让同一主题下，不同风格模式生成的图片差异更明显。

具体目标如下：

- 建立一组固定主题、可复跑、可对照的风格隔离测试。
- 强化不同 `模板风格` 在提示词层的专属表达，减少跨风格的模糊重叠。
- 在不大改 UI 和节点参数面的前提下，压低随机主题池对风格测试结果的干扰。

## 非目标

本次不包含以下内容：

- 不重做整个随机主题池结构。
- 不一次性扩充所有风格模板或重写所有主题库。
- 不先新增用户可见的复杂参数面。
- 不以“更花哨”替代“更清晰的风格分化”。

## 设计原则

- 先隔离变量，再优化风格。
- 先验证提示词层是否足够分化，再判断是否需要新增模式。
- 风格差异优先体现为视觉语言差异，而不是主体身份、职业、场景或道具的大幅漂移。
- 新测试必须可复跑，便于后续回归。

## 方案概述

### 1. 新增风格隔离测试矩阵

新增一组“同主题、不同风格”的测试用例，固定以下维度：

- 主体
- 场景
- 构图
- 动作
- 技术画质
- 额外约束

只切换以下维度：

- `模板风格`
- 风格 profile 导出的 `style_lead`
- 风格专属 `style_tags`
- 风格导向 `extra_fragments`

测试分为两轮：

1. `运行时随机标签 = false`
   用于验证纯风格 profile 的基础分化。
2. `运行时随机标签 = true`
   但测试路径内尽量约束主体和场景漂移，只观察风格层是否仍然稳定成立。

输出物至少包括：

- 每个风格的最终提示词
- 对照图片
- 汇总报告，便于横向比较

### 2. 强化风格 profile

重点优化 `stage_prompt/prompt_builder.py` 中以下结构：

- `_RUNTIME_STYLE_VARIANT_PROFILES`
- `_RUNTIME_STYLE_TRACK_OVERRIDES`
- `_RUNTIME_STYLE_IDENTITY_OVERRIDES`
- `_RUNTIME_STYLE_SCENE_CUES`

优化方向：

- 让 `style_lead` 更像真正的风格起始词，而不是泛化质量词。
- 让 `style_tags` 更专属，减少不同风格共享同一批模糊标签。
- 让 `extra_fragments` 更偏视觉导向，少用抽象修辞，多用能影响构图、材质、光色和成片气质的语言。

预期结果：

- `真实感` 更像摄影体系内部的差异。
- `插画感` 更像绘制媒介、笔触、动画语义与画面组织的差异。
- `CG感` 更像渲染管线、材质结构和空间表现的差异。
- `古风` 更像东方服饰、场景语义与气韵表达的差异。
- `神话感` 更像仪式性、奇观性、史诗性与神性叙事的差异。

### 3. 控制随机主题池对风格测试的干扰

本次不直接扩大 UI 参数，而是在测试与内部逻辑层优先控制干扰。

优先做法：

- 在测试用例层固定核心主题标签。
- 在测试路径中限制 `_RUNTIME_DIVERSITY_VARIANT_PROFILES` 对主体与场景的大幅替换。
- 优先观察纯风格层增强后的效果。

如果测试结果表明“风格层增强已经有效，但随机主题池在真实运行中仍显著稀释差异”，则可作为后续工作继续评估是否引入一个正式的“风格优先”运行时模式。

## 代码范围

主要涉及以下文件：

- `stage_prompt/prompt_builder.py`
- `stage_prompt/randomizer.py`
- `run_template_example_suite.py`
- `tests/fixtures/regression_cases.json`
- 必要时新增一个专门的风格矩阵脚本或测试夹具

## 测试与验收

### 验收标准

通过线定义如下：

- 同一主题下，不同 `模板风格` 生成的提示词前导与风格标签能稳定分化。
- 对照图片能形成可辨识的风格簇，而不是仅在职业、场景或道具上变化。
- 新增测试可重复执行，并能作为后续风格回归基线。

### 失败信号

以下情况视为本次优化无效或不达标：

- 不同风格的提示词仍大量复用同类模糊词。
- 图片差异主要来自题材变化，而不是视觉风格变化。
- 某一风格增强后明显污染其他风格的常用语义。
- 测试结果只能靠人工主观猜测，缺少稳定对照。

## 风险与边界

主要风险：

- 过度拉开风格差异，导致某些风格模式提示词变得刻板或模板化过强。
- 为了制造差异而污染主体层，最终让风格和题材绑定过死。
- 随机主题池的漂移仍可能掩盖局部风格增强效果。

控制方式：

- 保持主体、场景和风格三层职责清晰。
- 优先改风格层，不先扩大随机层改动面。
- 通过固定主题矩阵验证改动是否真正提升风格辨识度。

## 实施顺序

1. 建立风格隔离测试矩阵与输出报告。
2. 跑当前基线，记录不同风格的提示词与图片对照。
3. 定向优化风格 profile。
4. 复跑测试，对比优化前后效果。
5. 评估是否需要后续补一个正式的“风格优先”运行时模式。
